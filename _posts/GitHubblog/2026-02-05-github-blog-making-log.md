## 잔디가 안 심어진 이유 찾기

처음에는 블로그 저장소에 커밋을 해도 잔디가 심어지지 않아 당황했다.

회사 컴퓨터에서 작업해서
컴퓨터나 이메일 연동이 안 되어 그런 줄 알았지만,
다른 저장소를 fork 할 때마다 잔디가 찍히는 것을 보니
그 문제는 아닌 것 같았다.

그래서 계속 원인을 찾아보던 중,

> fork한 템플릿 기반 저장소는
> 포스트를 작성해도 잔디가 안 찍힐 수 있다

는 내용을 발견했다.

---

## 테스트 저장소로 확인

이를 확인하기 위해 테스트용 저장소를 만들어 직접 실험했다.

테스트 저장소에는 잔디가 정상적으로 찍혔고,
블로그 저장소에는 찍히지 않았다.

즉,

* 계정의 잔디 시스템은 정상
* 일반 커밋도 정상 집계됨
* 문제는 블로그 저장소만 해당됨

이라는 결론을 얻었다.

---

## Test Repo 만드는 방법

### 1. 저장소 생성

* GitHub 우측 상단 `New repository`
* 예시 이름: `test-contribution`

### 2. 설정

* `Public` 선택
* `Add a README file` 체크
* `Create repository` 클릭

### 3. 커밋 생성

* `README.md` 열기
* 우측 상단 `Edit` 버튼 클릭
* 내용 아무거나 추가
  예: `잔디 테스트용 커밋입니다.`

Commit message 예시:

```text
test commit
```

* `Commit directly to the main branch` 선택
* `Commit changes` 클릭

### 4. 결과 확인

* 잔디가 찍히면 → 계정 정상
* 안 찍히면 → Contribution 설정 문제

---

## 블로그 저장소에서 잔디가 안 찍히는 이유

### 1. Pages 저장소 커밋은 기여로 덜 잡히는 경우

다음과 같은 커밋은 잔디에서 누락될 수 있다.

* 설정 파일 수정 (`_config.yml`)
* 테마 파일 복붙
* 자동 생성 파일
* GitHub 웹에서 생성된 일부 커밋

---

### 2. 커밋 Author / Email 정보 문제

블로그 repo 커밋에는 이메일 줄이 제대로 표시되지 않았다.

즉,

* GitHub는 커밋 자체는 인정하지만
* Contribution Graph에는 연결하지 않는 상황이었다.

---

### 3. GitHub Actions / Pages 배포 커밋

블로그는 내부적으로

* build
* deploy

자동 작업이 섞여 있어
봇 커밋처럼 처리되기도 한다.

---

## 해결 방법

### Fork 연결 끊기

```text
Settings > General > Leave fork network
```

원본 템플릿 업데이트는 받을 수 없지만
잔디는 정상적으로 심어진다.

---

또 다른 방법으로는

> 복사한 커밋을 새로운 레포지토리로 옮기는 방법

도 있었지만,
나는 그냥 마음 편하게 fork 연결을 끊었다.

---

## Fork 연결을 끊은 이유

* 원본 업데이트가 몇 년 전부터 멈춤
* 이미 완성형 테마라고 판단
* 굳이 업데이트 받을 필요가 없다고 생각함
